# 演策 · 问题定义（v2）

## 1. 最终交付物

一个**生产级 AI 周易决策推演产品**：

- **前端**：surge.sh 静态部署（已有，React + Three.js）
- **后端**：Railway 部署 Node.js 服务（LLM 代理 + Agent 编排 + 数据持久化）
- **核心体验**：输入问题 → 演（主Agent）分析问题 → 动态召唤智囊 → 多视角辩论 → 周易算法起卦 → 生成可编辑天命牌 → 保存/分享 → 社区互动
- **产品完整度**：所有页面真正可用，社区可互动，数据可持久化，作为一个真正的AI产品运营

## 3. 核心设计原则

### 3.1 动态 Agent 编排（非固定8个）

**现状问题**：当前固定8个Agent + 关键词匹配，每次都浮现相同数量，不符合实际需求。

**目标**：
- **"演"作为主Agent（Orchestrator）**：接收用户问题后，先调用 LLM 分析问题特征
- **动态决定召唤哪些视角的Agent**：简单问题（如"1+1"）可能只需1个；复杂人生抉择可能需要5-6个
- **Agent可扩展**：不限于当前8个，后续可添加更多视角（如法律、健康、教育等）
- **每次浮现效果不同**：Agent数量、组合、出场顺序基于问题内容动态生成
- **异常处理**：LLM 分析失败时降级到关键词匹配；Agent召唤数量异常（0个或太多）时有限制逻辑

### 3.2 真正融入周易思想

**现状问题**：当前卦象是问题类型→卦象的固定映射，不是真正的周易。

**目标**：
- **真实起卦算法**：大衍之数（50策→49→分二→挂一→揲四→归奇）或金钱卦（三枚铜钱六次）
- **卦象基于算法结果**：不是固定映射，而是真实算出的六爻
- **卦辞/爻辞取自《周易》原文**：64卦384爻完整数据
- **变爻处理**：有变爻时取变爻辞，无变爻时取卦辞
- **五行生克**：结合日辰天干地支判断旺衰

### 3.3 LLM 提供商优先级（已调整）

| 优先级 | 提供商 | 模型 | 用途 | 原因 |
|--------|--------|------|------|------|
| 1 | 智谱 | glm-4-flash（免费） | 主力 | 免费、中文能力强 |
| 2 | 魔搭 | 开源模型 | 备选 | 免费、开源 |
| 3 | DeepSeek | deepseek-chat | 兜底 | 质量高但收费，最终降级用 |

## 4. 隐含要求

### 4.1 产品完整度
- **所有页面真正可用**：首页、推演台、每日运势、剧本、卡牌册、日历、词典、社区——每个都有真实功能，不是空壳
- **社区可互动**：用户可发帖、回复、点赞，不只展示静态数据
- **数据联动**：推演记录→日历可见、卡牌册→社区可分享、每日签到→运势刷新

### 4.2 AI产品维度
- **Agent全栈技术**：后端Agent编排、并发调用、超时降级、流式输出
- **运营**：用户留存机制（签到、成就、社区互动）、分享传播（天命牌截图）
- **设计**：统一视觉系统、加载动画、错误状态、空状态
- **法务**：免责声明（决策辅助非预测）、用户协议、隐私政策
- **安全**：API限流、输入过滤、XSS防护、数据加密

### 4.3 天命牌
- **生成模板**：推演完成后自动生成精美命签卡牌
- **可编辑**：用户可修改命签上的文字（标题、感悟、承诺）
- **可保存**：存到后端，换设备可查看
- **可分享**：生成分享图，可发到社区或社交媒体

## 6. 最容易翻车的风险点

| # | 风险 | 影响 | 缓解方案 |
|---|------|------|---------|
| 1 | "演"分析问题失败/返回异常 | Agent无法召唤 | 降级到关键词匹配；限制Agent数量1-6个 |
| 2 | LLM 响应慢/超时 | 用户等待流失 | 并发调用 + 流式输出 + 8s超时降级 |
| 3 | Agent数量不稳定（0个或太多） | 体验异常 | 硬限制：最少1个，最多6个 |
| 4 | 周易算法实现错误 | 卦象不准 | 用《周易》原文验证，单元测试 |
| 5 | 智谱免费额度用完 | 服务降级 | 自动切换DeepSeek，本地预设兜底 |
| 6 | 社区功能数据存储复杂 | 开发周期长 | 先做基础CRUD，后续迭代 |
| 7 | 移动端3D性能 | 卡顿/崩溃 | 检测移动端，降级为2D视图 |
