# 演策 · 方案选择

## 已确认架构方向：方案A — Railway Node.js Express 独立后端

### 技术栈最终选择

| 层 | 技术 | 理由 |
|----|------|------|
| 前端框架 | **Vite + React 19**（保留） | 现有代码成熟，3D推演台已运行，不需要迁移 |
| 3D渲染 | **React Three Fiber + Three.js**（保留） | React生态最好的3D方案，3A游戏用的是Unreal/Unity，但Web端R3F就是最优解 |
| 2D动画 | **Framer Motion**（保留） | 与React深度集成，现有动画已成熟 |
| 后端框架 | **Node.js + Express** | 轻量、生态成熟、Railway原生支持 |
| 数据库 | **Railway PostgreSQL** | 自带、免费、SQL稳定 |
| LLM主力 | **智谱 glm-4-flash**（免费） | 中文强、免费额度大 |
| LLM兜底 | **DeepSeek** | 质量高，收费但最终降级用 |
| 部署 | **surge.sh（前端）+ Railway（后端）** | 均免费、国内可访问 |

### 为什么不换 Next.js / 不用游戏引擎

| 方案 | 不选的原因 |
|------|-----------|
| Next.js 全栈 | Three.js SSR 兼容问题是已知坑，3D组件需 `'use client'` 包裹，迁移风险不可控 |
| Babylon.js | 比 Three.js 重，生态不如 R3F，现有代码要全部重写 |
| Unreal Pixel Streaming | 需要GPU服务器，月费$100+，不符合免费约束 |
| Cocos Creator | 面向游戏非Web应用，React生态不兼容 |

**结论**：R3F + Three.js 就是 Web 端 3D 交互的最优解。问题不在框架，在于当前代码是**固定流程写死的**，要改成**数据驱动**。

## 核心架构设计

### 1. 后端架构（Railway Node.js Express）

```
演策后端
├── /api/agent/analyze     ← "演"分析问题，返回需要哪些Agent
├── /api/agent/dialogue    ← 指定Agent生成回应（SSE流式）
├── /api/divination/cast   ← 周易起卦算法（大衍之数/金钱卦）
├── /api/divination/interpret ← 卦象解读（LLM生成）
├── /api/cards/*           ← 天命牌CRUD
├── /api/community/*       ← 社区帖子/回复/点赞
├── /api/daily/*           ← 每日运势
├── /api/calendar/*        ← 推演日历
└── /api/achievements/*    ← 成就系统
```

### 2. 动态 Agent 编排引擎（核心创新）

**流程**：
```
用户输入问题
    ↓
后端 /api/agent/analyze
    ↓
调用智谱 LLM，system prompt:
  "你是'演'，分析用户问题，决定需要哪些视角的顾问。
   可选视角：财务、职业、风险、情感、反思、宏观、行动、沟通、法律、健康...
   返回JSON：{ agents: [{id, name, stance, persona}], reasoning: '...' }"
    ↓
返回Agent列表（1-6个，动态）
    ↓
前端接收列表 → 3D场景动态渲染Agent位置（根据数量算360°分布）
    ↓
前端逐个请求 /api/agent/dialogue → SSE流式返回对话
    ↓
对话在3D场景中以打字机效果浮现
```

**异常处理**：
- LLM分析失败 → 降级到关键词匹配（现有逻辑）
- 返回0个Agent → 强制至少1个（镜渊·反思视角）
- 返回超过6个 → 截取前6个
- Agent对话超时(8s) → 降级到本地预设
- 所有LLM失败 → 本地预设完成全流程

### 3. 前端3D交互改造

**现状问题**：
- [Game.jsx](file:///d:/Appt/大三下/Trae赛2/sandbox-app/src/pages/Game.jsx) 用固定 setTimeout 控制流程（4s→7s→9.5s...）
- [AgentGhosts.jsx](file:///d:/Appt/大三下/Trae赛2/sandbox-app/src/components/board/AgentGhosts.jsx) 渲染固定8个Agent
- [layoutConfig.js](file:///d:/Appt/大三下/Trae赛2/sandbox-app/src/components/board/layoutConfig.js) 的 `getAgentPosition` 已经支持动态数量

**改造方向**：
- Game.jsx 从固定setTimeout → 改为**事件驱动**（后端返回数据后触发下一阶段）
- AgentGhosts 从固定列表 → 改为**接收动态Agent列表**渲染
- Agent浮现动画：每个Agent从光球中"诞生"→飞到自己的八卦位置→开始对话
- 对话文字：3D空间中用 `Html` 组件跟随Agent位置，打字机效果
- "演"分析问题时：光球脉动加速，卦象粒子旋转加速
- Agent出场顺序：后端返回的顺序，不是前端写死的

### 4. 真实周易起卦

**金钱卦算法**（三枚铜钱六次）：
```
每次掷三枚铜钱：
- 正面2枚 + 反面1枚 = 少阳（─ ─ 阳爻）
- 正面3枚 = 老阳（— — 变阳爻）
- 反面2枚 + 正面1枚 = 少阴（— — 阴爻）
- 反面3枚 = 老阴（─ ─ 变阴爻）

六次从下往上排 → 得到本卦 + 变卦
有变爻 → 取变爻辞
无变爻 → 取卦辞
```

**数据**：64卦完整卦辞、爻辞、彖传、象传（从《周易》原文录入）

### 5. 天命牌系统

**生成**：推演完成后，后端根据卦象+对话+用户选择生成命签数据
**模板**：前端用精美的水墨风卡片渲染
**编辑**：用户可修改标题、个人感悟、承诺文字
**保存**：存到后端 PostgreSQL
**分享**：生成分享图（html2canvas），可发到社区

### 6. 社区系统

**数据模型**：
- `posts`：用户分享的命签 + 文字
- `replies`：回复
- `likes`：点赞
- 用户身份：匿名（基于localStorage生成唯一ID）或简单昵称

**功能**：
- 发帖（分享命签）
- 浏览（按热度/时间排序）
- 回复
- 点赞
- 按标签筛选（职业/情感/财务/人生...）
