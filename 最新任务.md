## 一、现状诊断（4 大问题）

### 问题 1：演问了问题，但用户没有回答窗口 ❌

**代码层已有但未生效**：
- `Game.jsx` 第 231 行 `setAwaitingUser(true)` 后，`handleUserAdvance` 第 247-282 行会保存 `currentResponse` 作为对演的回答，并调 `streamYanChat` 让演提炼信息注入 Agent context
- **但 UI 层 `AgentDialogueOverlay` 在 `yan_analyze` 阶段没有渲染输入框**，导致用户只看到演的发言却无法回答
- 后果：演拿不到用户的详细信息 → 无法把关键背景传给各 Agent → Agent 发言泛泛

**修复方向**：
1. `AgentDialogueOverlay.jsx` 在 `phase === 'yan_analyze' && awaitingUser` 时渲染输入框 + 继续按钮
2. 输入框样式：浮在演发言下方，水墨风格，placeholder「写下你的回答…」
3. 用户回答后：演 LLM 提炼成 2-3 句「用户背景摘要」→ 注入 `inference.userContext` → 后续 Agent 都能看到

### 问题 2：Agent 是真 Agent 还是 prompt 限制的 LLM？ ❌（核心问题）

**当前真相：是 prompt 限制的 LLM，不是真 Agent**

| 维度 | 真 Agent 应有 | 演策现状 | 差距 |
|------|--------------|----------|------|
| 自主决策 | 自己决定追问/调用工具/结束 | 状态机硬编码推进 | ❌ 无自主性 |
| 工具调用 | 查日历/股价/政策/搜索 | 无任何工具 | ❌ 完全没有 |
| 记忆 | 跨会话记忆 + 工作记忆 | `multiAgentFramework.js` 有 `AgentMemory` 类但**未启用**；前端有 `memoryStore.js` 但只文本存储 | ⚠️ 框架有，未接入 |
| 协作 | 消息传递 + 互相反驳/引用 | `previousDialogues` 单向注入（后续 Agent 看前面发言） | ⚠️ 伪协作 |
| 状态管理 | 每个 Agent 有独立状态 | 全局 `inference` 对象共享 | ❌ 无独立状态 |
| 角色卡 | persona + 工作方法 + 交付标准 | 只有 persona | ⚠️ 单层提示词 |

**关键发现**：`src/services/multiAgentFramework.js`（470 行）已经实现了 `Agent` 类（带 memory/ask/followUp/judgeContinueAsking）+ `AgentTeam` 类（turnOrder/processTurn/generateSummary），**但 Game.jsx 完全没用它**，而是用 `inferenceEngine.js` 的函数式调用。

### 问题 3：多 Agent 处理什么？技术深度够吗？怎么防抄袭？

**多 Agent 应该处理的**：
1. **信息收集分工**：财务 Agent 算账、风险 Agent 查负面证据、职业 Agent 看赛道
2. **辩论收敛**：正方/反方/法官三角，3-5 轮硬上限
3. **决策建议**：每个 Agent 给出立场 + 置信度，演仲裁收敛

**当前技术深度**：浅。护城河 = 0。任何人 copy prompt + 调 LLM 就能复刻。

**防抄袭的五大护城河**（需逐个落地）：
1. **用户习惯壁垒**：每日卦签培养「遇决策先起卦」心智
2. **数据飞轮**：命签反馈闭环 + 用户决策画像积累 → 越用越懂用户
3. **复合系统复杂度**：析问→辩论→占卜→命签→收藏→复卦→历史 全链路深耕
4. **垂直专精**：专注「重要决策辅助」，积累卦象解读库 + 决策案例库
5. **切换成本**：命签收藏 = 用户情感 + 决策档案，越久越不愿离开

### 问题 4：CC / WorkBuddy 的 harness/tool/skill/memory/agent.md 怎么学？

**Claude Code (CC) 四层架构 → 演策转化**：

| CC 层级 | CC 机制 | 演策转化方案 |
|---------|---------|-------------|
| 记忆层 | `CLAUDE.md` 四级作用域（用户/项目/会话/任务），启动全量加载 | 建 `演.md` 四级：用户画像级 / 问题领域级 / 推演会话级 / 当前阶段级 |

|----------------|----------|

**关键洞察**：同一模型在不同 Harness 下表现差异 >> 不同模型在同一 Harness 下差距。**所以我们的发力点不是换更强 LLM，而是把 Harness 做厚。**

## 五、调研学习成果（已转化为上述方案）

### 5.1 Claude Code (CC) 四层架构 — 实际实现细节

**CLAUDE.md 四级作用域**：
| 作用域 | 位置 | 加载时机 |
|---|---|---|
| 组织级 | `/Library/Application Support/ClaudeCode/CLAUDE.md` | 会话开始全量 |
| 项目级 | `./CLAUDE.md` 或 `./.claude/CLAUDE.md` | 会话开始全量 |
| 用户级 | `~/.claude/CLAUDE.md` | 会话开始全量 |
| 子目录级 | `<subdir>/CLAUDE.md` | **按需加载**（读到该子目录才加载） |
- 配套 `.claude/rules/*.md` 用 frontmatter `paths` 字段做路径作用域规则
- 推荐 ≤200 行，越长遵循度越低
- `@import` 语法引用其他文件
- 压缩后根 CLAUDE.md 重新注入，子目录的会丢

**Skills 渐进式披露**：
- `SKILL.md` frontmatter：`description`（常驻触发器）/ `disable-model-invocation` / `user-invocable` / `context: fork`（独立 subagent）/ `allowed-tools` / `paths`
- 会话开始只加载 name+description（几十 token），被触发才加载完整 body
- 动态注入 `` !`cmd` `` 在喂给模型前先执行 shell 并替换为输出
- 四级存储：Enterprise > Personal(`~/.claude/skills/`) > Project(`.claude/skills/`) > Plugin

**Hooks 三层嵌套**：
- 配置 `settings.json` 的 `hooks` 键：`事件名 → matcher group → handler 数组`
- 20+ 事件：SessionStart/UserPromptSubmit/PreToolUse/PostToolUse/Stop/PreCompact 等
- Handler 四种类型：command（shell）/ http / prompt / agent
- **Exit code 语义**：0=允许，1=非阻塞错误，**2=阻塞拒绝**
- Matcher 粗筛 + `if` 细筛两阶段过滤
- `{retry: true}` 让模型自我修正

**MCP（Model Context Protocol）**：
- JSON-RPC 2.0，三种传输：stdio / HTTP / SSE
- 配置 `.mcp.json`，六级优先级（企业独占 > 本地 > 项目 > 全局 > 插件 > 远程）
- 工具命名 `mcp__<server>__<tool>`，和内置工具同构
- 环境变量展开 `${VAR}` / `${VAR:-default}` 避免密钥明文
- Elicitation 双向交互（server 可反向问用户）

### 5.2 WorkBuddy / AutoGen / CrewAI / Swarm — 多 Agent 协作实际实现

**三层提示词结构**（CrewAI + juejin 实战）：
- `identity`（身份锚定）：角色 + 核心价值观 + 红线（显式列出不可逾越）
- `methodology`（工作方法）：按序执行步骤
- `deliverable`（交付标准）：硬约束，违反即重生成
- 用 XML 标签 `<identity>...</identity>` 包裹比 Markdown 边界信号更强

**结构化消息总线**（MetaGPT Blackboard 模式，演策应选这个）：
```javascript
// 消息结构
{ id, agentId, role, round, content, confidence, references:[msgId], msgType }
// msgType: claim(表态) / rebuttal(反驳) / support(补充) / question(追问) / verdict(裁决)
// 三种模式：广播式(AutoGen) / 订阅式(MetaGPT) / P2P式(AutoGen send)
// 演策选订阅式：Agent 只看自己关心的消息类型
```

**交接文件（Handoff Packet）**：
```json
{
  "handoff_id": "ho_xxx",
  "from_phase": "yan_analyze",
  "to_phase": "agent_debate",
  "task_context": { "question", "question_type", "user_background" },
  "artifacts": { "yan_analysis", "team_composition", "blackboard_snapshot[最近10条]" },
  "constraints": { "max_rounds":2, "max_tokens_per_turn":200, "must_reference_previous":true },
  "expected_output": { "format":"json", "schema":{...} }
}
```

**token 三层预算**：
- 会话级：`max_round`（演策建议 3 轮）
- Agent 级：`max_consecutive_auto_reply`（连续自动回复上限）
- 调用级：`maxTokens`（已做到 200）
- 中文 token 估算：汉字数 × 1.5 + 其他字符 × 0.3（不能用 `len()`）
- 软上限 70% 触发压缩，硬上限 95% 强制截断

**辩论收敛（Wald SPRT 自适应停止）**：
- 3 个收敛信号任一触发即停：轮次硬上限 / 共识分≥0.8 / 循环检测相似度>0.85
- 简化版 SPRT：共识分>0.7 累积 `+log(3)`，<0.4 累积 `+log(0.33)`
- 上界 `log(0.9/0.05)≈2.89` → 收敛停止
- 下界 `log(0.1/0.95)≈-2.25` → 不收敛停止
- 80% 问题 1-2 轮即收敛，比固定 3 轮省 30%+ 调用

### 5.3 关键洞察汇总

1. **Harness > 模型能力**：同一 GLM-4-Flash 在好 Harness 下远胜裸调，发力点是做厚编排不是换 LLM
2. **演策不需要引入新框架**：激活已有 `multiAgentFramework.js` 骨架 + 补齐 `Blackboard` / `DebateConvergence` / `TokenBudget` 三个类即可达"真 Agent"标准
3. **确定性 vs 概率性分离**：CLAUDE.md/skills 是"建议"（模型可能不遵守），hooks 是"确定性"（每次必跑）。安全护栏必须用 hooks
4. **消息 ≠ 对话历史**：结构化消息（带 agentId/role/round/confidence/references/msgType）才是真协作的基础
5. **description 是触发器，body 是载荷**：Skill 常驻的只有 description，body 按需加载，可装几十个 skill 不撑爆上下文
